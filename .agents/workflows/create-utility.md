# Workflow: Create Utility

This workflow defines the step-by-step process and rules to follow when designing and implementing a new utility function or library in the codebase. It integrates rules from `.agents/rules/` to ensure architectural integrity, type safety, documentation standards, and localized compliance.

## Phase 2: Architectural Placement & Types

1. **Directory Placement:**
   - Store global utilities under `/src/utils/` or `/src/lib/`.
   - Store module-specific utilities within the immediate folder of the module they serve.
2. **Type Safety:**
   - Write fully typed TypeScript functions. 
   - Avoid the use of `any` or `@ts-ignore`. Declare custom interfaces/types if needed.
   - If the utility integrates with Payload CMS, make sure to use types generated by payload (via `bun run generate:types`) to ensure strict E2E type safety.
3. **Imports Standard:**
   - Always import packages and files relative to the project root using `@/` path aliases.

---

## Phase 3: Internationalization (i18n) & Formatting

1. **Native Localization:**
   - If the utility formats numbers, dates, lists, or currency, **NEVER** use manual string manipulation or custom formatting libraries.
   - **MUST** use the native JavaScript `Intl` API (`Intl.NumberFormat`, `Intl.DateTimeFormat`, etc.) for Persian (fa) localization.
   - Ensure the calendar system defaults to Jalali/Shamsi and numbers render in Persian numerals when stylistically appropriate.
2. **Text Processing:**
   - If processing user text, handle RTL text boundary issues and Arabic/Persian character normalization (e.g. converting Arabic Kaf/Yeh to Persian Ke/Ye) correctly.

## Phase 4: Documentation (MANDATORY)

### 1. TSDoc Standard (MANDATORY)
- Document the utility using strict **TSDoc** syntax according to the guidelines in [.agents/rules/documentation.md](file:///d:/Scripts/sampleshop/.agents/rules/documentation.md).
- **English Language Requirement:** All TSDoc comments, inline comments, and markdown documentation files (such as `README.modules.md` and `ARCHITECTURE.md`) MUST be written exclusively in **English**, regardless of the application's locale or target language (e.g., Persian/fa).
- You MUST document:
  - **Generics (`@typeParam`):** Write an individual tag for *every single* generic type parameter. Explain its constraints, purpose, and usage.
  - **Inputs (`@param`):** Write an individual tag for *every single* input parameter. Detail the parameter's name, type, constraints, and semantic description.
  - **Defaults (`@defaultValue`):** Write an individual tag for *every* optional parameter or configuration option to specify its default value.
  - **Outputs (`@returns`):** Write a detailed description of the return type, its meaning, range of possible values, and conditional return scenarios.
  - **Exceptions (`@throws`):** Write an individual tag for *each* specific error that can be thrown, describing the conditions under which it occurs and potential recovery steps.
- Focus on the "Why" and details of edge cases (e.g. null/undefined inputs, negative numbers, overflow boundaries).

### 2. Localized Algorithmic Documentation (`README.modules.md`)
- Create or update the `README.modules.md` file in the utility's immediate folder.
- Must include:
  1. **Algorithmic Breakdown:** Detailed step-by-step explanation of the internal logic.
  2. **Data/Execution Flow:** A Mermaid diagram showing the flow of data or execution paths (especially for complex parsing, validations, or mathematical models).
  3. **Consumer Tracking:** A list of absolute file paths that import this utility.
  4. **Integration Graph:** A Mermaid flowchart showing which high-level components or routes consume this module.
  5. **Dependencies:** External libraries or package requirements.

### 3. Global Architecture Map (`ARCHITECTURE.md`)
- If the utility changes global data flow or introduces new external dependencies, update the high-level flowchart in the root `ARCHITECTURE.md`.

## Phase 5: Unit Testing (MANDATORY)

1. **Test Generation:**
   - Every utility must have a corresponding unit test file (e.g., `[utility-name].test.ts`) in the same directory.
2. **Framework Compliance:**
   - Read `package.json` to identify the framework (this project uses **Vitest**).
   - Write tests covering standard success paths, boundary cases, error handling, and invalid inputs.
3. **Test Run:**
   - Do NOT run tests manually during utility creation, as they are executed automatically via `pre-commit` hooks. For manual verification/testing, run the `/verify-build` workflow.

---

## Phase 6: Verification & Cleanup

1. **Verification Gate:**
   - Do NOT manually run build checks, linting, or tests. These verification steps are automatically run via `pre-commit` hooks during staging/commit/push.
   - If manual verification or build fixing is requested, run the `/verify-build` workflow.
2. **Code Cleanup:**
   - Ensure no debug statements or mock structures are left behind without a `// TODO: REMOVE BEFORE PRODUCTION` flag.
3. **Versioning:**
   - Run `bun version patch` (or `minor`/`major` where appropriate) on completed features. Let the user include this in their commit.

---

## Phase 7: Handoff & User Commits

1. **Handoff:**
   - Present the completed work clearly to the user, highlighting the files created or modified.
   - Do NOT run git commit or git add commands, and do not block progress waiting for commits. Leave all staging, committing, and branching/pushing operations entirely to the user.
   - (Optional) You may suggest appropriate conventional commit messages for their reference, but do not execute them.
